# NVB Notion Schema

This document is the single source of truth for all Notion database schemas.
When setting up databases manually, match every property name and type exactly.

**The Notion property names in this doc must match the constants in `packages/schema/src/notion-properties.ts`.**

---

## Database 1: NVB Inbox

**Purpose:** Every incoming request enters here. The inbox is the primary state machine for NVB. Status on each item tracks where it is in the pipeline.

**How to connect:** In Notion, share this database with your integration via the "Connect to integration" option (top-right "…" menu on the database).

| Property Name | Type | Options / Notes |
|---|---|---|
| Title | title | Short description of the request. Auto-set by system. |
| Request ID | rich_text | `NVB-XXXXXXXX` — unique identifier generated by intake. |
| Request Type | select | `idea`, `claim`, `task`, `question`, `repo_task`, `other` |
| Raw Request | rich_text | Full unedited user input. Max 2000 chars stored (Notion limit). |
| Status | **status** | See status options below. |
| Priority | select | `low`, `medium`, `high`, `critical` |
| Workflow | select | `verdict` (for now — reserved for future workflow types) |
| Human Decision | select | `Pending`, `Approve`, `Revise`, `Reject`, `Needs Evidence` |
| Confidence | number | 0.0–1.0. Float. Set by clerk after verdict run. |
| Latest Run | relation → NVB Runs | One-to-one link to the most recent run page. |
| Final Artifact | relation → NVB Artifacts | Set after promotion. |
| Source Links | url | First source URL provided by user. (Notion URL type = 1 URL) |
| Notes | rich_text | Submitter or reviewer notes. |
| Created At | date | Set by system at intake. |

**Status options (in order):**

| Status | Meaning |
|---|---|
| Queued | Request received, pipeline not started |
| Running | Pipeline actively processing |
| Awaiting Review | Verdict complete, waiting for human decision |
| Approved | Human set Approve — ready for promotion |
| Revised | Human requested revision — re-queue for another run |
| Rejected | Human rejected — archived |
| Promoted | Artifact has been created |
| Archived | Soft-deleted / no longer active |

**Category groups for the Status property:** Use Notion's status type (not select) so you get the built-in color grouping. Suggested groups:
- Not started: Queued
- In progress: Running, Awaiting Review
- Done: Approved, Promoted
- Action needed: Revised, Needs Evidence (under Human Decision)
- Closed: Rejected, Archived

## Database 2: NVB Evidence

**Purpose:** Every source, citation, or piece of evidence retrieved or generated during a pipeline run is stored here. This is the research trail.

| Property Name | Type | Options / Notes |
|---|---|---|
| Title | title | Short label for this evidence item. |
| Inbox Item | relation → NVB Inbox | The inbox item this evidence belongs to. |
| Evidence Type | select | `Web`, `Doc`, `Code`, `Claim`, `Counter`, `User Provided` |
| Source URL | url | Direct URL to the source. |
| Source Ref | rich_text | Short citation string (e.g. "MDN Web Docs, 2024"). |
| Excerpt | rich_text | Relevant quote or excerpt from the source. Max 2000 chars. |
| Relevance | rich_text | Why this evidence matters for the request. |
| Confidence | number | 0.0–1.0. How confident the retriever is in this source. |
| Retrieved By | select | `generator`, `auditor`, `user` |
| Notes | rich_text | Free-text notes. |

## Database 3: NVB Runs

**Purpose:** Full audit log of every pipeline execution. One row per run. Stores model, prompts used, timing, exit status, and the raw verdict JSON.

| Property Name | Type | Options / Notes |
|---|---|---|
| Title | title | Auto-generated: `{run_type} — {datetime} UTC` |
| Inbox Item | relation → NVB Inbox | The inbox item this run processed. |
| Run Type | select | `generator`, `auditor`, `clerk`, `full` |
| Model | rich_text | Model name used (e.g. `gpt-4o`). |
| Prompt Version | rich_text | Prompt version string: `gen:v1.0.0 aud:v1.0.0 clk:v1.0.0` |
| Started At | date | ISO timestamp — pipeline start. |
| Finished At | date | ISO timestamp — pipeline end. |
| Exit Status | select | `approve`, `revise`, `reject`, `needs_evidence`, `error` |
| Confidence | number | Final synthesized confidence from clerk. |
| Error Summary | rich_text | Set only if Exit Status = error. |
| Raw JSON | rich_text | Full verdict JSON (truncated to 2000 chars; full version in page body). |
| Notes | rich_text | Manual notes. |

## Database 4: NVB Artifacts

**Purpose:** Registry of all promoted, approved outputs. Each artifact is a real deliverable — a Notion page with structured content.

| Property Name | Type | Options / Notes |
|---|---|---|
| Title | title | `Artifact: {original request title}` |
| Source Inbox Item | relation → NVB Inbox | The inbox item that was promoted. |
| Artifact Type | select | `doc`, `issue`, `spec`, `summary`, `none` |
| Status | select | `Draft`, `Published`, `Archived` |
| Summary | rich_text | First 2000 chars of unified_answer. |
| Notion URL | url | Direct link to this artifact page. Set after creation. |
| GitHub URL | url | GitHub issue/file URL. Null if not created. |
| Published At | date | When the artifact was created. |
| Notes | rich_text | Manual notes. |

**Important:** The actual content (full unified_answer, citations, notes) is written to the page body as Notion blocks — not in properties. This is intentional. Notion property limits (2000 chars for rich_text) would truncate most useful artifacts.

## Optional Database 5: NVB Prompts

Add this **only if** you want to version and manage prompts from Notion rather than from code.

| Property Name | Type | Notes |
|---|---|---|
| Title | title | `{role} — {version}` e.g. `Generator — v1.0.0` |
| Role | select | `generator`, `auditor`, `clerk` |
| Version | rich_text | Semver: `v1.0.0` |
| System Prompt | rich_text | The full system prompt text. |
| Active | checkbox | Only the active version is used at runtime. |
| Notes | rich_text | Changelog notes for this version. |

**Verdict:** Skip this for MVP. Prompt versioning in code (packages/prompts/src/*.ts) is simpler and testable. Add NVB Prompts only if you want non-technical reviewers to edit prompts.

## Relation map

```
NVB Inbox ←──── NVB Evidence (many-to-one via Inbox Item)
NVB Inbox ←──── NVB Runs    (many-to-one via Inbox Item)
NVB Inbox ←──── NVB Artifacts (one-to-one via Source Inbox Item)
NVB Runs  ──── NVB Inbox (via Latest Run backlink)
NVB Artifacts ──── NVB Inbox (via Final Artifact backlink)
```

---

## Setup checklist

1. Create 4 databases in a shared Notion page (or workspace root)
2. Add every property from this doc — exact names, exact types
3. Create an internal integration at https://www.notion.so/profile/integrations
4. Connect the integration to each database (via database "…" menu → Connect to integration)
5. Copy each database ID from the URL and add to `.env`
6. Run `pnpm validate-env` to confirm access
